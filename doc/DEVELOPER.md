# Developer Guide

This repository is a small personal Emacs configuration. It is not intended to
be a general Emacs distribution, a Doom Emacs clone, or a full programming IDE.

The main goal is to keep Emacs useful as an Org/GTD, writing, and lightweight
navigation tool while keeping the configuration understandable in one sitting.

## Project Direction

The owner primarily uses Emacs for:

- Org mode task management
- GTD review and capture
- Markdown and prose writing
- Reading or lightly editing files
- Desktop coordination with beorg on iPhone/iPad

The owner does not primarily use Emacs for coding. Avoid drifting this project
toward LSP, debugger, tree-sitter, modal editing, or large IDE-style workflows
unless explicitly requested.

The desired style is closer to:

> Small, native Emacs, Org-focused, predictable.

It should not become:

> A broad framework with many optional subsystems.

## Repository Layout

```text
~/.emacs.d/
├── init.el                    # Minimal bootstrap; loads config.org
├── early-init.el              # Early startup settings
├── config.org                 # Main literate configuration source
├── config.el                  # Generated by org-babel; ignored by Git
├── user-settings.example.el   # Template for local settings
├── user-settings.el           # Local private settings; ignored by Git
├── beorg-init.sample.org      # Sample beorg mobile init.org
├── README.md                  # User-facing quick start
└── doc/DEVELOPER.md           # This file
```

Only `config.org` should be treated as the source of truth for Emacs behavior.
Do not edit `config.el` by hand.

## Configuration Model

Startup is intentionally simple:

1. `early-init.el` disables package auto-activation before regular init.
2. `init.el` runs `org-babel-load-file` on `config.org`.
3. `config.org` loads `user-settings.el` if present.
4. Missing user variables fall back to defaults.
5. Package declarations and workflow configuration are evaluated from
   `config.org`.

Personal machine details belong in `user-settings.el`, not in `config.org`.
Examples:

- `my/org-dir`
- `my/font`
- `my/font-size`
- `my/theme`
- `my/org-extra-agenda-files`

## Org Workflow

The configuration is organized around a small GTD file set derived from
`my/org-dir`:

```text
inbox.org      # unprocessed capture inbox
tasks.org      # all clarified action items
ideas.org      # low-friction thoughts and notes
```

The intended flow is:

```text
Capture -> inbox.org -> clarify -> refile to tasks.org -> agenda review
```

Core Org behavior:

- `C-c c` opens `org-capture`.
- `C-c a` opens `org-agenda`.
- `C-c g i/t/e` opens the core Org files.
- `C-c g d` opens the daily agenda dashboard.
- `C-c g n` opens the inbox for processing.

The GTD/Things mapping is:

```text
Inbox     -> inbox.org
Today     -> agenda daily view
Upcoming  -> agenda 14-day view
Anytime   -> unscheduled NEXT tasks
Someday   -> SOMEDAY tasks in tasks.org
Personal  -> personal tag
Work      -> work tag
Logbook   -> CLOSED/LOGBOOK on completed tasks
Ideas     -> ideas.org for low-friction thoughts
```

TODO states are intentionally small:

```text
TODO -> NEXT -> DONE
WAITING/SOMEDAY -> CANCELLED
```

Tags are similarly constrained:

```text
personal work home errands computer health learning travel energy_high energy_low quick
```

When extending the Org system, prefer improving this workflow over adding new
parallel systems.

`tasks.org` is initialized with:

```org
* Tasks
```

Personal/work/health and similar dimensions should be expressed as tags, not
separate files.

## beorg Integration

The iOS app beorg is used as a mobile companion for the same Org directory.

`beorg-init.sample.org` is a sample file for the beorg-side `init.org`. To use
it, copy it into the synced Org directory as lowercase `init.org`.

The expected mobile role is:

- view agenda
- quick capture
- light edits
- simple state changes

The expected desktop Emacs role is:

- inbox processing
- refiling
- larger edits
- weekly review

Because Dropbox sync can create conflicted copies when both sides edit the same
file, avoid simultaneous edits from Emacs and beorg. Prefer using beorg mostly
for quick capture into `inbox.org`.

## Package Policy

Packages should earn their place by supporting the daily workflow.

Current package themes:

- `vertico`, `orderless`, `marginalia`: completion
- `consult`: search and navigation
- `embark`, `embark-consult`: actions on completion candidates
- `which-key`: key discovery
- `modus-themes`: stable built-in-feeling theme
- `org-superstar`: Org readability
- `markdown-mode`: writing
- `dired-sidebar`: lightweight file tree
- `exec-path-from-shell`: macOS PATH integration

Avoid adding packages for:

- purely cosmetic changes
- heavy programming language environments
- modal editing experiments
- broad frameworks that hide simple Emacs behavior

When adding a package:

1. Add it to `my/packages`.
2. Configure it with `use-package`.
3. Guard optional behavior with `:if` or fallback helpers.
4. Keep related config in the same section of `config.org`.

## Keybinding Policy

This configuration uses native Emacs keybindings. It deliberately does not use
Evil/Vim modal editing.

Rules for new bindings:

- Keep standard Emacs and Org keys working.
- Put personal global commands under `C-c`.
- Put GTD file/workflow shortcuts under `C-c g`.
- Prefer mnemonic bindings over clever ones.
- Do not introduce a leader-key system unless the owner explicitly asks for it.

Important current bindings are documented in `README.md`. Avoid maintaining a
separate exhaustive keymap document unless it is generated or clearly needed.

## Runtime Files

Emacs naturally creates runtime files in `.emacs.d`. These are not part of the
source configuration and should generally remain ignored by Git:

- `config.el`
- `bookmarks`
- `recentf`
- `projectile-bookmarks.eld`
- `auto-save-list/`
- `auto-saves/`
- `backups/`
- `elpa/`

The repository should stay small even if the working directory contains these
runtime files.

## Maintenance Principles

Prefer small, direct changes.

Good changes:

- make capture faster
- make agenda clearer
- reduce Dropbox/beorg conflict risk
- simplify package or file layout
- improve documentation accuracy
- keep macOS behavior predictable

Risky changes:

- turning this into an IDE
- replacing native keybindings with modal editing
- adding a package for one rare command
- introducing new abstraction layers before there is real duplication
- editing generated files instead of source files

Before finishing a behavior change, run at least:

```sh
emacs --batch -l init.el
```

For documentation-only changes, review the rendered Markdown/Org text enough to
catch stale references and wrong file names.

## Common Tasks

Install missing packages inside Emacs:

```text
M-x my/install-missing-packages
```

Regenerate tangled config manually:

```text
M-x org-babel-tangle
```

Open the main config:

```text
C-c g c
```

Open the inbox:

```text
C-c g i
```

Daily review:

```text
C-c g d
```

## Non-Goals

Do not optimize for these unless explicitly requested:

- language servers
- code completion frameworks
- debugger integration
- terminal multiplexing
- project-wide build systems
- Git porcelain inside Emacs
- Doom-style module declarations
- broad custom framework macros

The best future version of this repository is still simple.
